Why Supabase + Vercel?

Stack Comparison

Vibe coding was already decided. The question is which services to build on. Four realistic stacks emerged — all viable for AI-assisted development, but only one gives the right balance of power and simplicity.

✓

Supabase + Vercel — the chosen stack

Database: PostgreSQL via Supabase (auth, storage, realtime built-in). Hosting: Vercel (best Next.js DX). Why it wins: PostgreSQL supports schemas, views, and RLS — the foundation of our architecture. Supabase gives enterprise DB features without ops overhead. Vercel deploys are instant. Microsoft SSO via NextAuth. Free tier generous, scales to ~$170/mo.

~

Azure PostgreSQL + Vercel — viable, but more work

Database: Azure Database for PostgreSQL (Flexible Server). Hosting: Vercel or Azure Static Web Apps. Why viable: Same PostgreSQL as Supabase — schemas, views, RLS all work identically. GDPR-compliant with EU regions. Good if you have Azure credits or compliance requirements. Why not chosen: Lose Supabase's batteries (auth, storage, realtime). You build auth via NextAuth, storage via Azure Blob + your own API. More infrastructure, less AI-friendly patterns. Cost similar (~$30-50/mo).

~

Firebase + Vercel — popular but wrong data model

Database: Firestore (NoSQL). Hosting: Vercel. Why not: NoSQL can't do the complex joins, views, or schema isolation our architecture requires. Firestore is document-based — great for chat apps, wrong for integrated business systems. We'd end up fighting the data model. Firebase Auth is solid but doesn't change the verdict.

✗

Railway — all-in-one simplicity, less flexibility

Database + Hosting: Combined platform. Why not: Tighter coupling between DB and hosting means vendor lock-in. Less mature ecosystem than Vercel for Next.js. Combined billing makes cost allocation harder. If we need to move hosting later, the DB moves too. Separation of concerns (Supabase for data, Vercel for compute) gives us more optionality.

Requirements that narrowed the field
🗄️

PostgreSQL, not NoSQL

Schema-per-app isolation, SQL views for integration, Row Level Security — these aren't optional. Only PostgreSQL-based stacks survived.

🔗

One database, many apps

15+ apps reading from shared master data (employees, clients, projects). Needs proper relational constraints and transactional integrity.

🛡️

GDPR compliance for HR data

Both Supabase and Azure offer EU regions + DPA. Either works. We still build GDPR features (data export, deletion) in apps regardless of provider.

🏢

Microsoft SSO non-negotiable

All stacks support it via NextAuth or similar — not a differentiator, but eliminated options without OAuth flexibility.

⚡

Batteries included > build it yourself

Supabase gives auth, storage, realtime out of the box. Azure PostgreSQL is just a database — you build everything else. For vibe coding, fewer moving parts wins.

Governance

No dedicated technical lead — oversight is distributed across 2–4 active builders. Claude Code Review does the deep technical analysis; humans verify the findings. The goal is light enough that nobody abandons it.

The Safety Net: Protection That Doesn't Depend on Process

Safety

This is the answer to the governance question. Database-level protection operates regardless of which governance model is running, regardless of who reviewed the code, and regardless of what Claude generated.

Always on — no process required
🔒

Schema isolation + role-based access

Each builder's database role can only write to their own app's schema. Dropping a table in another app is physically impossible — the database rejects it. No review process can catch everything; this does.

🛡️

Row-Level Security (RLS)

Database silently adds per-user filters to every query at the engine level. Even if Claude generates a query that forgets to filter by user, the data never leaks. Overhead: 1–5%, negligible for internal tools.

🌍

Dev / production separation

Builders physically cannot connect to the production database — they don't have the connection string. It lives only in Vercel's env vars and GitHub Actions secrets. The worst a bad deploy can do is break the app; data is safe.

📋

Atlas migration linting

Every database migration is automatically checked before it runs. DROP COLUMN, DROP TABLE, and other destructive operations are flagged and blocked. Runs in the CI pipeline — no human has to remember to check.

💾

Automated backups + historization

Supabase runs daily point-in-time backups (7-day retention on paid tier). Critical tables use PostgreSQL temporal tables to track changes — every edit creates a history record. Can restore to any minute or audit who changed what when.

Added by governance (layer on top)
🤖

Claude Code Review (automated, every Pull Request)

Application-level issues that the database can't catch: missing auth checks, business logic errors, vulnerable dependencies. Catches 1.7× more issues than human-only review of AI-generated code.

👁️

Human peer review (shared responsibility)

Reads Claude's findings, checks the Vercel preview, approves or requests changes. Does not require technical expertise — the AI surfaces what needs attention.

💡 This is why lighter governance is viable. Schema isolation, RLS, and dev/prod separation protect against the most dangerous failure modes — data loss, cross-app corruption, accidental production changes — regardless of who reviewed the code. The governance layer catches quality and logic issues, not catastrophic ones.

Gateway Implementation Options

Deep Dive

Two viable approaches for insider.addvery.com — both work with this architecture. Choice depends on team coordination preferences and UX priorities.

📂 Option A: Directory Start here
Lightweight app: handles auth and navigation only. Users log in, see a grid of apps relevant to their role, click through to each app's own URL.
  • Each app is a separate Next.js project, deployed independently
  • App registry lives in the core DB: name, URL, icon, allowed roles
  • Role access is a simple app_access(app_id, role_id) table
  • Microsoft SSO tokens shared across apps on the same domain
  • Gateway itself rarely changes once set up — trivially simple
  • Downside: navigation between apps feels like separate page loads
🏢 Option B: Umbrella shell Grow into this
Shared shell with persistent sidebar, top bar, and notifications — the Microsoft 365 portal pattern. All apps in one Next.js monorepo as route groups sharing layout and auth session.
  • Seamless navigation — no page reloads between apps
  • Consistent design system enforced by shared layout
  • Downside: all apps deploy together — change in one redeploys all
  • More complex to coordinate across multiple builders
  • Better UX once the app portfolio is mature
  • Micro-frontends can preserve independent deploys but add complexity
✅ Non-negotiable regardless of option: authentication + role-based access from day one. First version is just three tables: apps, roles, app_access + Microsoft SSO.

Database Views & RLS: How It Actually Works

Deep Dive

SQL examples showing how apps share data through views and how Row-Level Security prevents data leaks even when code has bugs.

View Example: Invoicing reads Timesheets
CREATE VIEW timesheets.v_billable_hours
AS SELECT
te.project_id,
p.client_id,
SUM(te.hours) AS total_hours,
p.billing_rate
FROM
timesheets.time_entries te
JOIN
core.projects p USING (project_id)
WHERE
te.approved = true
AND te.billed = false
GROUP BY
te.project_id, p.client_id, p.billing_rate;
-- Invoicing app just reads this view.
-- If Timesheets changes internal tables,
-- they update the view. Invoicing never breaks.
💡 Views are the integration layer. No APIs, no webhooks, no message queues — just SQL.
RLS Example: Users only see their own data
ALTER TABLE timesheets.time_entries
ENABLE ROW LEVEL SECURITY;
CREATE POLICY user_own_entries ON
timesheets.time_entries
FOR ALL USING (
auth.uid() = user_id
OR auth.has_role('manager')
);
-- Even if Claude generates bad code:
SELECT * FROM time_entries;
-- PostgreSQL silently adds:
WHERE user_id = current_user_id()
-- Data never leaks. Engine-level protection.
🛡️ RLS runs at the database engine level with 1-5% overhead — negligible for internal tools. Even buggy code can't leak data.

Migration History: Your Database Evolution Audit Trail

Deep Dive

Every schema change lives in Git as a timestamped migration file. This creates a permanent, auditable record of how the database evolved — critical for compliance, onboarding, and preventing schema drift.

📁 supabase/migrations/ (in Git)
20260101120000_create_core_schema.sql
20260108093000_create_timesheets_schema.sql
20260115141500_add_approval_workflow.sql
20260122160000_create_invoicing_schema.sql
20260205103000_add_billable_view.sql
-- Every change = one file, timestamped
-- Git tracks: who, when, why (commit msg)
-- Can recreate DB from scratch: run in order
Example: Understanding a Mystery Column
New builder asks: "Why does invoices table have a reconciliation_status column?" You don't remember. Git blame → migration file → commit message: "Added for quarterly audit reconciliation per Finance team request (ticket #892)." Full context, instantly.
What Migration History Gives You
  • ✓ Audit trail: Compliance can see who changed what when
  • ✓ Onboarding: New builders read migrations to understand schema
  • ✓ Disaster recovery: Restore from backup, replay migrations
  • ✓ Schema drift detection: Compare dev vs prod migrations
  • ✓ Rollback safety: Know exactly what each migration did
How It Works in Practice
1. Claude generates a migration
ALTER TABLE timesheets.time_entries
ADD COLUMN overtime_approved BOOLEAN;
2. Saved as timestamped file
20260320_add_overtime_approval.sql
3. Committed to Git with context
git commit -m "Add overtime approval
field to support new labor law
compliance requirements"
4. Applied to dev, then prod via CI
supabase db push
✓ Migration applied
# Prod happens on merge to main
What This Prevents
Schema drift: "Prod has columns dev doesn't" becomes impossible — migrations are the single source of truth. Weekly supabase db diff confirms they match. No manual SQL in prod ever.
💡 Migration files are immutable once merged. Never edit old migrations — write a new one. This preserves the timeline. Git history + migration files = complete database evolution story.

Dev / Prod Database Separation: Your Safety Baseline

Deep Dive

Two completely separate Supabase projects — one for building and breaking things, one for real user data. This separation is the foundation of every other safety measure.

🔧 Development Database
Purpose: Builder playground. Test features, run migrations, break things freely.
What lives here:
  • • Fake/seed data only
  • • Test users, test projects, test invoices
  • • Every builder has full access
  • • Reset anytime with supabase db reset
Connection string in:
.env.local (each builder's machine)
DATABASE_URL=supabase-dev-...
Why This Matters
Builders can experiment without fear. Bad migration? supabase db reset and you're back to clean state in 30 seconds. This removes the psychological barrier to trying things.
🚀 Production Database
Purpose: Real user data. Changed only via CI/CD pipeline after approval.
What lives here:
  • • Real employee data, real timesheets
  • • Real invoices, real expenses
  • • NO builder has direct access
  • • Changed ONLY via merged Pull Requests
Connection string stored in:
Vercel environment variables (secret)
GitHub Actions secrets (for CI/CD)
NOT in .env files, NOT on builder machines
What This Prevents
Accidental data loss: Builder can't accidentally run a destructive migration against prod — they physically can't connect. Testing gone wrong: Even if a builder runs a delete script locally, it's hitting test data only. Schema experiments: Try a new column structure in dev first, see if it works, then promote to prod via Pull Request.
✅ This is governance tier 0 — it works regardless of whether you have peer review, automated checks, or full DevOps. Even with zero process, this separation alone prevents the worst disasters.
